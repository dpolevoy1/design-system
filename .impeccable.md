# Assemble UI Impeccable Guidelines

## Positioning

Assemble is a decision-support intelligence layer, not a generic chat shell. The interface must consistently bias toward:

- shorter time-to-decision
- visible confidence and provenance
- progressive control for expert operators
- low-friction execution from insight to action

This system serves multiple surfaces (core web app, Knowledge, Admin, Metric Validation, and PowerPoint add-in) while preserving one visual and interaction language.

---

## Product Context and Users

### Primary Users

- Account Executives, CRO, and operations stakeholders using Assemble as a daily decision layer.
- They operate in high-context workflows, usually mid-task, with low tolerance for decorative UI.
- They require reliable hierarchy: what changed, what matters now, what action to take next.

### Secondary Users

- Team admins and analysts validating quality, permissions, and data behavior.
- New users onboarding through guided states and progressively disclosed complexity.

### UX Intent by Surface

- `New Question`: fast input loop with context and tools as secondary affordances.
- `Post-Meeting Insights`: triage and review-first table patterns.
- `Knowledge`: source health, confidence signaling, and file governance.
- `Admin Dashboard`: operational control with efficient table actions.
- `Metric Validation`: analytical workflow with dense but legible information architecture.
- `PowerPoint add-in`: focused, constrained side panel flows optimized for task completion.

---

## Brand and Interaction Tone

### Personality

- Authoritative, precise, calm.
- Utility-first over ornamental.
- Premium through spacing, typography, and motion quality, not visual noise.

### Voice Rules

- Outcome-first copy on system surfaces.
- No cheerleading language or "assistant fluff".
- Microcopy should be explicit, short, and operational.
- Errors should be direct and action-oriented.

---

## Core Design Principles

1. **Decision velocity over decoration**  
   Every element should reduce cognitive overhead or improve confidence.

2. **Token-first consistency**  
   Do not hardcode ad hoc colors/spacings when a design token exists.

3. **Progressive disclosure by default**  
   Advanced controls appear contextually (hover, focus, open state), not all at once.

4. **State clarity is mandatory**  
   Hover, focus, selected, disabled, warning, error, success must be visually unambiguous.

5. **Cross-surface coherence**  
   Core primitives (buttons, inputs, badges, tables, sidebar behavior) must feel like one system across all pages and the PowerPoint add-in.

6. **Accessibility and keyboard parity**  
   Focus visibility, readable contrast, and keyboard support are baseline requirements, not enhancements.

7. **Composable over bespoke**  
   Prefer stable primitives that compose well in dense product surfaces over one-off visual inventions.

### Color Rules

- Primary blue (`600`) is the main interactive anchor.
- Orange is reserved for destructive/warning-flavored action emphasis where specified.
- Red should indicate failure/error semantics; avoid decorative red usage.
- Badge and status color usage should remain low-area and semantically scoped.
- Neutral ramps should carry most layout and table scaffolding.
- Status color must never be the only source of meaning (pair with icon/text/label).
- `primary-300` and `primary-400` are preferred hover/focus background tints on Assemble surfaces.
- `primary-100` and `primary-200` are preferred subtle panel/background tints for PowerPoint-specific contexts.
- `secondary-600` is the default icon tone in neutral contexts; `secondary-700` is the selected icon tone.
- `neutral-400-list-outline` is the default outline token for list/table container boundaries.

## Typography

- Prioritize clear hierarchy over stylistic variation.
- Headline, section labels, and table headers should remain compact and high-contrast.
- Body text should stay readable in dense table and form layouts.
- Avoid introducing alternate decorative type scales without explicit design-system approval.
- Keep table-heavy pages on a restrained typographic rhythm to reduce scan fatigue.

## Spacing and Rhythm

- Maintain strict vertical rhythm in long operational views.
- Keep action clusters and filters visually grouped by function.
- Use whitespace to separate decision blocks, not to create decorative emptiness.

## Motion and Timing

Use quiet, utility-focused motion. Motion should explain state changes, not draw attention to itself.

- Rail and container transitions: smooth ease curves with moderate duration.
- Row and action-state transitions: short and consistent.
- Hover feedback: immediate but low-amplitude.
- Reduced motion mode must preserve full usability and state clarity.

Do not combine multiple unrelated animations on one interaction path.

## Component Rules (Source of Truth: Design System)

When a screen variant conflicts with the component library, use the design-system behavior and tokens listed here.

## Buttons

Supported families:

- Primary
- Secondary
- Tertiary (where applicable)
- Delete
- PowerPoint-specific button variants

Supported structures:

- icon-only
- text-only
- icon + text

Supported sizes:

- default (`h38`)
- small (`h32`)

Required states:

- default
- hover
- focus
- disabled

Rules:

- Keep state transitions subtle and consistent.
- Do not invent one-off visual styles in feature screens if equivalent component variants exist.
- For icon-only buttons, ensure hit area and focus visibility are preserved.
- Keep semantic usage stable:
  - Primary for principal confirm/submit flow.
  - Secondary for adjacent but non-primary actions.
  - Tertiary for low-emphasis utility actions.
  - Delete for destructive operations only.
- Trigger actions on release, not on press-down (activate only when press and release complete on the same target).
- Button labels must be explicit about outcome; avoid vague labels such as `OK`.
- Use sentence case for multi-word labels (capitalize the first word only).
- Do not allow two-line button labels; widen the button or rewrite the label.
- Do not abbreviate labels to force fit.
- Do not use ellipsis in action labels; rename the action to clarify non-final steps when needed.
- Use `6px` as the default horizontal spacing between adjacent CTA buttons.
- In dense action toolbars, keep icons only for essential actions.
- Icon-only actions are allowed only when meaning is immediately recognizable.

## Inputs

Supported modes include:

- base and small
- default/hover/focus/disabled
- filled
- warning
- error
- success
- with clear action icon
- paired/inline input patterns where specified

Rules:

- Placeholder, value, and validation text hierarchy must stay consistent.
- Validation states must be visible without relying on color alone.
- Search inputs and selector inputs should reuse canonical input shells and icon placement.
- Input suffix/prefix actions (clear, dropdown, search icon) must align to canonical spacing.
- Do not collapse validation text spacing; reserve space for messages to avoid layout jump.
- Placeholder text must not prescribe a concrete value; use it as a hint, not as prefilled content.
- If the system already has a computed value, prefill it as actual value text (not placeholder-styled pseudo values).
- If the system assumes a value, prefill it as editable user value.
- Do not use placeholder text to describe format when mask behavior exists; use field masks for structured formats.
- Include a clear-action icon for fields users frequently reset.
- Keep 3-4 standard field widths per screen to preserve rhythm and scanability.
- Align field label baseline with entered text baseline; for multiline labels/fields, align to first line baseline.
- Field width should communicate expected value length (avoid arbitrary oversized fixed widths).
- Keyboard behavior: Enter may move focus to next field in forms, but must not auto-shift focus to submit from the last field.
- Focus-ring standard: blue focus (`#1745F7`) unless warning/error state owns the field until the user edits value.
- Error-style baseline: red border (`#FF3B4B`) with explicit inline error text.

## Breadcrumbs

Use breadcrumbs only for true multi-level navigation contexts.

Rules:

- Deep hierarchy contexts are valid.
- Shallow contexts should not force breadcrumbs if they add noise.
- Current node should be identifiable through styling and/or structure.
- Use breadcrumbs to show current position in hierarchy and provide quick return to parent levels.
- Breadcrumb separators are structural, low-emphasis markers; links carry hierarchy emphasis.
- Parent items are interactive links; the current page is a label state, not an active link.
- Breadcrumb behavior should stay consistent across top-navigation and side-navigation contexts.

## Checkbox

Required states:

- default
- hover
- focus
- disabled
- disabled selected
- selected
- partial selection
- completed
- error

Rules:

- Group labeling and alignment must remain clear in dense forms/tables.
- Focus ring and state feedback should remain visible on all backgrounds.
- Keep checkbox interaction areas practical in list rows (row-level click target where context allows).
- Prefer one-column checkbox groups; avoid multi-column checkbox matrices unless strongly justified.
- Partial selection state is reserved for hierarchical/group selection, not generic "in-between" semantics.
- Completed and error visual states must be semantically unique and never reused decoratively.

## Toggle

Required permutations:

- on/off
- default/hover/focus/disabled

Rules:

- Toggle placement should follow semantic grouping (prefer consistent side alignment).
- Avoid using toggles for irreversible/destructive operations.
- The label must communicate effect immediately; users should not infer side effects from context alone.
- Toggle changes apply immediately; do not require an extra `Save` step for a toggle change.
- Toggle supports on/off and interactive states, but should not carry warning/error validation semantics.
- If an option is unavailable, use a disabled state rather than introducing custom blocked patterns.
- In multi-toggle lists, prefer control on the left of label for faster vertical scanning consistency.

## Sidebar

Sidebar behavior and styling should follow the implemented shell patterns from PR `#704`.

### Structural Behavior

- Expanded rail width: `300px`
- Collapsed rail width: `68px`
- Collapsed mode is icon-first and label-suppressed
- Sub-navigation and secondary chat sections collapse cleanly in narrow rail

### Motion and Timing

- Rail transition uses smooth cubic-bezier timing for expand/collapse asymmetry
- Navigation row state transitions are slightly slower than "instant hover" to avoid jitter
- Collapse/expand should not cause icon "jumping" or mid-transition realignment.

### Surface and Theme

- Sidebar background derives from primary token tinting (`Primary-200` + subtle mix)
- Header receives a divider/surface treatment when scrolled
- Scrollbar styling remains thin and low-contrast
- Dark-mode token overrides are explicit and scoped

### Interaction Rules

- Collapsed logo has a dedicated hit area and visible focus ring
- Hidden labels in collapsed state must not intercept pointer events
- Hover/selected feedback uses low-opacity token-based backgrounds
- Collapsed rail should suppress non-essential metadata and keep icon scanline clean.

## Table and Data-Dense Patterns

Most Assemble surfaces are table-centric. Preserve predictable scanning behavior.

Required patterns:

- Sticky or persistent header hierarchy where applicable.
- Explicit sort affordances on sortable columns.
- Search and filter controls aligned and grouped before data rows.
- Row actions grouped consistently (do not scatter across random columns).
- Badge/status semantics consistent across Knowledge, Admin, and Post-Meeting tables.

Avoid:

- Oversized row heights without information gain.
- Icon-only actions with ambiguous meaning.
- Hidden destructive actions without clear affordance.

## Screen Implementation Guidance

Screens may contain legacy component styling. Normalize to the design-system components above.

## New Question

- Keep central input loop primary.
- Feature actions (knowledge/tools/improve prompt) are secondary and compact.
- Sidebar variant (expanded/collapsed) must not disrupt content hierarchy.
- Empty state and history state should share the same interaction grammar and control placement.

## Post-Meeting Insights

- Review queue and table affordances should be clear and scannable.
- Badge semantics and row-level actions should remain consistent with system primitives.
- Search and filters should align with canonical input/dropdown components.
- "Needs review" should be visually distinct from "resolved/healthy" without adding extra visual clutter.

## Knowledge

- Surface confidence and source status in a compact, legible way.
- Table controls should be practical and consistent: sort, filter, row actions.
- Distinguish "needs review", "healthy", and confidence categories clearly.
- Banner/health modules should remain informative at collapsed and expanded states.

## Admin Dashboard

- Prioritize high-throughput user management tasks.
- Keep table actions dense but readable (edit/delete/icon controls).
- Search and role filtering should remain top-level and quick to scan.
- Role/status badges should stay semantically stable with the rest of the platform.

## Metric Validation

- Preserve analytical density while protecting hierarchy.
- Complex validation banners/accordions should use reusable patterns, not bespoke styling.
- Do not let decorative gradients weaken data legibility.
- Expand/collapse behavior must preserve context and avoid disorienting content jumps.

---

## PowerPoint Add-in UI Guidance

PowerPoint is a constrained panel environment, not a full web canvas.

Rules:

- Prioritize compact layout and clear single-path actions.
- Reuse PowerPoint-specific button variants from the design system.
- Keep sign-in/onboarding flows lightweight and explicit.
- Footer/help text should remain concise and non-intrusive.
- Use add-in-safe spacing and touch targets for small panel widths.
- Treat panel height as constrained: avoid deep nested sections before the primary action.

---

## States, Feedback, and Validation

- Every interactive control must support keyboard-visible focus.
- Validation messaging must map to warning/error/success semantics.
- Destructive actions must be explicit and visually differentiated.
- Loading and empty states should preserve layout stability.
- Disabled states must remain legible and discoverable (not "invisible").
- Warning and error inputs must preserve state clarity during focus transitions.
- Success states should confirm completion without overpowering primary flow.

## Accessibility Baseline

- Keyboard traversal must reach all actionable controls.
- Focus indicators must be visible against all supported backgrounds.
- Information conveyed by color must have redundant encoding (icon, label, text).
- Ensure text contrast is acceptable in secondary/disabled contexts.
- Do not use hover-only disclosure for critical actions in keyboard-first flows.

---

## Implementation Guardrails

1. Prefer existing components and tokens over new custom UI.
2. If a screen still uses older styles, migrate to the design-system primitive.
3. Do not ship new hardcoded colors where token equivalents exist.
4. Keep component behavior consistent across app surfaces.
5. Preserve accessibility requirements during every visual update.
6. Keep app-shell and content-shell layering consistent with sidebar layout rules.
7. Avoid adding component variants unless the use case cannot be composed from existing primitives.

## Anti-Patterns (Do Not Ship)

- One-off button or input styling in feature screens.
- Decorative gradients or shadows that reduce data readability.
- Mixed state semantics (same color meaning different statuses per page).
- Inconsistent row action placement between similar tables.
- Focus styles removed or hidden for visual minimalism.
- "Quick fixes" that bypass tokens and introduce hardcoded palette values.

- [ ] Uses palette tokens (primary/secondary/neutral/additional) consistently.
- [ ] Sidebar behavior matches PR #704 interaction model.
- [ ] Focus, disabled, warning, error, success states are complete.
- [ ] Dense tables remain scannable and action-oriented.
- [ ] PowerPoint add-in screens keep compact, clear flow.
- [ ] No new one-off visual patterns without design-system justification.
- [ ] Feature screen legacy styles have been normalized where equivalent primitives exist.
- [ ] Reduced-motion behavior preserves usability.

## Quality Gate by Severity

- **P0**: broken navigation, inaccessible primary actions, destructive ambiguity.
- **P1**: incorrect component states, inconsistent semantic color usage, missing focus treatment.
- **P2**: spacing/hierarchy drift that slows scanning or introduces confusion.
- **P3**: cosmetic inconsistency without functional or comprehension impact.

---

## Non-Negotiables

- Design-system component behavior overrides legacy per-screen styling.
- Semantic color usage over decorative color usage.
- Operational clarity over ornamental complexity.
- Consistency across web and add-in surfaces is mandatory.
